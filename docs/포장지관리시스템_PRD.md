# 포장지 출력파일 관리 시스템 — PRD (v1: 재발주 프로세스)

작성일: 2026-07-22
근거 문서: [포장지_재발주_프로세스_AS-IS.md](./포장지_재발주_프로세스_AS-IS.md)
구축 방식: 자체 웹앱 개발 — 1단계 로컬 환경에서 구축·테스트 → 2단계 검증 후 외부 도메인 클라우드 서버로 이전

---

## 1. Problem Statement

비료·작물보호제 포장재(라벨, PP포대)의 재발주 프로세스가 전적으로 이메일에 의존하고 있다. 월 20건 이상 처리되는 상황에서 담당자(이정인 매니저·박현경 팀장·최효진 차장·조현종 소장)가 여러 건을 한 메일에 묶어 답신하는 경우가 많아 건별 이력이 누락되고, 어떤 파일이 최종 승인본인지 식별이 안 되며, 수정요청 이력이 부서별로 흩어져 있다. 이 상태가 지속되면 잘못된 버전으로 인쇄가 진행되는 리스크, 담당자의 반복적인 메일 관리 부담, 재발주 건 중복 진행이 계속 발생한다.

## 2. Goals

1. 재발주 프로세스의 모든 소통(요청·검토·수정요청·승인·반려)이 이메일 없이 시스템 내에서 완결된다.
2. 모든 재발주 건은 요청부터 완료까지 건 단위 타임라인으로 100% 이력 추적이 가능하다.
3. 최종 승인 파일의 오인식(잘못된 버전 사용)이 0건이 되도록 승인 파일이 자동 식별·잠금된다.
4. 담당자가 대시보드에서 자신에게 배정된 처리 대기 건을 실시간으로 확인할 수 있다.
5. 요청 등록부터 최종 승인까지의 처리 소요시간을 측정할 수 있는 데이터를 시스템이 축적한다 (베이스라인 확보 후 단축 목표 수립).

## 3. Non-Goals (v1)

- **신규 포장 디자인 제작 프로세스** — 기획·컨셉 단계가 포함되어 검토·승인 기준이 재발주와 다름. 데이터 모델(제품·파일·이력)은 재사용 가능하도록 설계하되, 워크플로우 자체는 v1 범위 밖.
- **단종(discontinuation) 처리 워크플로우** — v1은 재발주 흐름에 집중. 제품 상태값에 "단종" 여지를 남겨두되 전용 프로세스는 이후 단계에서 정의.
- **인쇄업체·ERP/발주 시스템과의 직접 연동** — 최종파일 전달까지가 v1 범위이며, 실제 발주·인쇄 시스템 연동은 별도 이니셔티브.
- **담당자(조직) 관리 콘솔의 완전한 셀프서비스화** — v1은 기존 4개 역할 + 대체 담당자 구조를 시스템에 반영하는 수준. 조직 개편에 따른 역할 신설/삭제 등 관리자 UI는 P2.
- **모바일 네이티브 앱** — 반응형 웹으로 대응하며, 카카오톡/문자는 어디까지나 알림 채널로만 사용(별도 앱 UI 없음).

## 4. User Stories

### 요청자 — 이정인 매니저 (울산공장)
- 재고 부족 또는 수정이 필요한 포장재를 제품명으로 검색해 재발주 건을 등록하고 싶다, 그래야 매번 파일을 찾아 첨부하지 않아도 된다.
- 내가 등록한 건이 지금 어느 단계(검토중/수정중/최종검수중/완료)에 있는지 한눈에 보고 싶다, 그래야 진행상황 확인차 메일을 쓰지 않아도 된다.
- 승인된 최종파일을 다운로드하거나, 시스템이 알려주는 최신 승인본을 그대로 인쇄에 사용하고 싶다, 그래야 잘못된 버전으로 인쇄하는 실수를 막을 수 있다.
- 동일 제품에 대해 이미 진행 중인 건이 있으면 등록 시 경고를 받고 싶다, 그래야 중복 재발주를 만들지 않는다.
- 여러 건의 최종 승인 파일을 한 번에 모아 다운로드하고 싶다, 그래야 인쇄 발주 시 파일을 하나씩 찾지 않아도 된다.

### 1차 검토·관리 창구 — 박현경 팀장 (브랜드기획팀, 부재 시 김신덕 본부장)
- 신규로 들어온 재발주 요청을 목록으로 확인하고 최종본 여부를 판단해 승인 또는 수정요청 처리를 하고 싶다, 그래야 이메일 없이 검토를 완결할 수 있다.
- 특정 파일에 대해 그동안 다른 채널로 접수된 오타·수정요청 메모를 기록해두고, 재발주 시 자동으로 보여주고 싶다, 그래야 판단 기준을 놓치지 않는다.
- 연구소에서 승인된 최종파일을 시스템이 자동으로 나에게 관리 대상으로 넘겨주길 원한다, 그래야 별도로 파일을 찾아 공장에 전달하지 않아도 된다.
- 내가 부재중일 때 김신덕 본부장이 동일한 권한으로 검토·승인 업무를 대신 처리할 수 있길 원한다, 그래야 업무가 지연되지 않는다.
- 기존에 보유하고 있던 포장지 파일들을 한 번에 시스템으로 이관하고 싶다, 그래야 제품 하나하나 수동으로 등록하지 않아도 된다.

### 수정 담당 — 최효진 차장 (디자인)
- 나에게 배정된 수정요청 건과 구체적인 수정 지시사항을 목록으로 확인하고 싶다, 그래야 메일함을 뒤지지 않아도 된다.
- 수정한 파일을 시스템에 업로드하면 자동으로 새 버전으로 기록되고 박현경 팀장에게 재검토 요청이 가길 원한다, 그래야 별도 확인 메일을 쓰지 않아도 된다.

### 최종 검수·반려 판단 — 조현종 소장 (연구소)
- 박현경 팀장이 이관한 최종검수 대기 건을 목록으로 확인하고, 승인/수정필요/반려 중 하나로 처리하고 싶다.
- 반려를 선택할 때는 반드시 사유를 입력해야 하며, 이 사유가 담당자들에게 그대로 전달되길 원한다, 그래야 왜 반려됐는지 다시 설명할 필요가 없다.
- 반려 처리 권한은 나(연구소)에게만 있어야 한다, 그래야 승인 기준이 흔들리지 않는다.

### 공통 (전체 역할)
- 내가 담당해야 할 액션이 생기면(신규 요청·수정요청·재검토·검수요청) 시스템 알림함, 이메일, 카카오톡/문자로 즉시 알림을 받고 싶다, 그래야 이메일함을 상시 확인하지 않아도 놓치지 않는다.
- 특정 건을 열면 요청부터 지금까지의 전체 이력(누가·언제·무엇을·어느 채널로)이 시간순으로 보이길 원한다.

## 5. Requirements

### Must-Have (P0)

**P0-1. 제품·파일 관리**
- 제품명 기준으로 포장재(라벨/PP포대)를 등록·검색할 수 있다.
- 하나의 제품에 여러 버전의 파일(AI + JPG)이 연결되며, 업로드마다 자동으로 버전 번호가 증가한다.
- 승인 완료된 파일은 "최종 승인본"으로 자동 태그되고 잠금(수정·삭제 불가) 처리된다.
- 승인본이 갱신되면 이전 승인본은 자동으로 이력(과거 버전)으로 전환된다.
- 수용 기준
  - [ ] 제품명으로 검색 시 해당 제품의 모든 파일 버전이 시간순으로 노출된다
  - [ ] 최종 승인본은 목록/상세 화면에서 시각적으로 구분(배지 등)된다
  - [ ] 승인본이 아닌 파일은 인쇄용으로 다운로드할 수 없거나 명확한 경고가 표시된다

**P0-2. 일괄 업로드/다운로드 (이력 포함)**
- 여러 제품·파일을 한 번에 업로드할 수 있다. 업로드 시 파일명 또는 매핑표(예: 제품명-파일 매칭 목록)를 기준으로 각 파일이 올바른 제품에 연결된다.
- 기존에 보유 중인 포장지 파일들을 초기 이관(마이그레이션)할 때도 동일한 일괄 업로드 기능을 사용하며, 이때 과거 버전·승인 이력이 있다면 함께 입력할 수 있다(예: 승인일, 승인자, 비고를 담은 메타데이터 파일 동반 업로드).
- 여러 건(제품)의 최종 승인 파일을 선택해 압축파일(zip) 형태로 일괄 다운로드할 수 있으며, 압축파일에는 각 파일의 이력 정보(버전, 승인일, 승인자, 제품명)를 담은 목록이 함께 포함된다.
- 수용 기준
  - [ ] 여러 파일을 한 번에 선택해 업로드하면 각각 올바른 제품·버전으로 등록된다
  - [ ] 매핑에 실패한 파일(제품 미매칭 등)은 업로드가 차단되지 않고 별도로 표시되어 담당자가 수동 매칭할 수 있다
  - [ ] 다중 건을 선택해 다운로드하면 zip 안에 파일과 함께 이력 요약(제품명·버전·승인일·승인자)이 포함된 목록 파일이 생성된다

**P0-3. 재발주 요청 등록**
- 요청자는 제품을 선택해 재발주 건을 등록한다(재고부족/수정필요 사유 선택).
- 등록 시 동일 제품에 진행 중인 건이 있으면 경고와 함께 기존 건으로 이동을 제안한다.
- 수용 기준
  - [ ] Given 동일 제품에 진행 중인 건이 있을 때, When 새 요청을 등록하면, Then 중복 경고가 표시되고 기존 건 링크가 제공된다

**P0-4. 승인 워크플로우 (상태 머신)**
- 상태값: `요청등록 → 1차검토중 → (수정필요 → 디자인수정중 → 재검토중 → 1차검토중) → 최종검수중 → (승인 | 수정필요 | 반려) → 완료`
- 반려·수정필요 처리 시 사유 입력이 필수이며, 사유는 해당 건의 이력에 영구 기록된다.
- 반려 액션은 연구소 역할 계정만 실행할 수 있다(권한 분리).
- 1차 검토·최종 승인 권한은 박현경 팀장/김신덕 본부장이 동일하게 갖는다(대체 담당자).
- 최근 3개월 이내 연구소 승인 이력이 있는 파일은 "최종검수 생략 가능" 배지가 표시되며, 박현경 팀장 승인만으로 완료 처리할 수 있다.
- 수용 기준
  - [ ] Given 조현종 소장이 반려를 선택했을 때, When 사유를 입력하지 않고 제출하면, Then 제출이 차단되고 사유 입력을 요구한다
  - [ ] Given 박현경 팀장 계정으로 로그인했을 때, When 반려 액션을 시도하면, Then 권한 없음으로 차단된다
  - [ ] Given 최근 3개월 내 승인 이력이 있는 파일일 때, When 박현경 팀장이 승인 처리하면, Then 연구소 검수 없이 완료 상태로 전환된다

**P0-5. 역할별 대시보드**
- 로그인한 역할(요청자/1차검토/디자인/연구소)에 따라 "내가 처리해야 할 건" 목록이 최상단에 노출된다.
- 건별 카드/행에 제품명, 현재 상태, 대기 경과일, 최근 조치자가 표시된다.
- 수용 기준
  - [ ] 각 역할 로그인 시 본인 액션이 필요한 건이 0건이어도 "처리할 건 없음" 상태가 명확히 표시된다
  - [ ] 대기 경과일이 임계치(예: 3일) 초과 시 시각적으로 강조된다

**P0-6. 건별 이력(타임라인)**
- 모든 상태 변경, 파일 업로드, 코멘트, 반려사유가 시간순 타임라인으로 기록된다.
- 각 이력 항목에는 조치자·소속·일시·채널(시스템/이메일 등)이 함께 기록된다.
- 수용 기준
  - [ ] 임의의 건을 열었을 때 요청 시점부터 현재까지의 모든 조치가 누락 없이 시간순으로 보인다

**P0-7. 알림**
- 상태 변경으로 새로운 액션이 필요한 담당자에게 시스템 인앱 알림 + 이메일(자동 발송, 수동 작성 없음)이 발송된다.
- 카카오톡/문자 알림은 최소 "신규 배정" "반려" 두 이벤트에 대해 발송된다.
- 수용 기준
  - [ ] 상태가 바뀌어 다음 담당자에게 액션이 넘어가면 1분 이내 인앱 알림이 생성된다
  - [ ] 반려 처리 시 관련자 전원에게 카카오톡/문자 알림이 발송된다

**P0-8. 대체 담당자 라우팅**
- 박현경 팀장을 "부재중"으로 설정하면 해당 기간 신규 배정 건이 김신덕 본부장에게도 동일하게 노출·처리 가능해진다.
- 수용 기준
  - [ ] 부재 설정 중에는 박현경 팀장과 김신덕 본부장 대시보드에 동일한 대기 건이 노출된다

### Nice-to-Have (P1)

- 오타·수정요청 메모를 파일/제품 단위로 상시 기록해두는 "비고" 기능(정식 재발주 외 채널로 들어온 요청 축적)
- 건별 코멘트/멘션 기능 (담당자 간 시스템 내 짧은 논의)
- 대시보드 내 월간 처리 건수·평균 소요시간 리포트
- 파일 비교(이전 버전 대비 변경사항 시각적 diff — 이미지 오버레이 등)
- 담당자 관리 화면(역할 배정, 부재 설정)을 관리자가 셀프서비스로 운영

### Future Considerations (P2)

- 신규 포장 디자인 제작 워크플로우 (기획→디자인→검토→승인 단계가 재발주보다 복잡)
- 단종 처리 워크플로우 및 제품 상태(단종) 관리
- 인쇄업체/ERP 발주 시스템 연동 (승인 파일 자동 전송)
- 조직 개편에 대응하는 완전한 역할·권한 관리 콘솔
- 다국어 지원

## 6. Success Metrics

**Leading (출시 후 2~4주)**
- 재발주 건 중 이메일로 이탈 없이 시스템 내에서 완결된 비율 — 목표 100%
- 담당자별 대시보드 일일 활성 사용률 — 목표 90% 이상 (4개 역할 기준)
- 신규 배정 알림 발생 후 24시간 내 확인율 — 목표 95% 이상

**Lagging (출시 후 1~3개월)**
- 건별 요청~완료 평균 소요시간 — 베이스라인 측정 후 목표치 수립(예: 30% 단축)
- 최종파일 오인식으로 인한 재인쇄/오류 신고 건수 — 목표 0건
- 반려 사유 기록률 — 목표 100% (누락 시 시스템이 차단하므로 사실상 보장)

## 7. Open Questions

- 카카오톡/문자 발송을 위한 발신 채널(알림톡 API, 문자 중계사 등)을 이미 계약해둔 곳이 있는가, 아니면 신규 계약이 필요한가? (담당: 사용자/IT)
- 파일 저장소는 사내 서버/NAS인가, 클라우드 스토리지(예: S3 계열)인가? 보안 정책상 제약이 있는가? (담당: 사내 IT)
- "최근 3개월 이내 승인" 예외 규칙의 3개월 기준일은 최초 승인일인가, 최근 승인일인가? (담당: 박현경 팀장)
- 로그인/계정 체계는 사내 SSO(있다면) 연동인가, 별도 계정 발급인가? (담당: 사내 IT)
- 대기 경과일 강조 임계치(예: 3일)의 적정 기준값은? (담당: 박현경 팀장)
- 기존 보유 파일을 일괄 이관할 때, 과거 승인 이력(승인일·승인자 등)을 어떤 형식으로 준비할 수 있는가(엑셀 매핑표 등)? 이력 정보가 없는 파일은 "이력 미상"으로 표시하고 최초 승인부터 새로 진행할 것인가? (담당: 박현경 팀장/이정인 매니저)
- 로컬 구축 단계의 실행 환경(사내 PC/서버 스펙, OS, 네트워크 접근 범위)은 어떻게 되는가? (담당: 사용자/사내 IT)
- 클라우드 이전 시 사용할 도메인·호스팅사·예산이 정해져 있는가, 아니면 이 단계에서 함께 정해야 하는가? (담당: 사용자/사내 IT)
- 클라우드 이전 시점에서 로컬 단계에 축적된 파일·이력 데이터의 마이그레이션 방식은? (담당: 엔지니어링)

## 8. Timeline Considerations

- 명시적 데드라인은 없으나, "빠른 출시보다 제대로 만드는 것"이 우선순위로 확인됨.
- 향후 신규제작·단종 프로세스 및 담당자관리 고도화가 예정되어 있으므로, v1 데이터 모델(제품/파일/버전/이력/역할)은 이 확장을 염두에 두고 설계할 것.
- **배포 단계가 2단계로 나뉜다**: (1) 로컬 환경에서 구축·기능 테스트 → (2) 검증 완료 후 외부 도메인을 가진 클라우드 서버로 이전. 로컬 단계에서부터 클라우드 이전을 염두에 두고 설계(파일 저장 경로 추상화, 환경변수 기반 설정 등)해 이전 시 재작업을 최소화할 것.
- 제안 단계: **1단계(본 PRD, 재발주 프로세스 시스템화 + 로컬 구축)** → **1.5단계(클라우드 이전)** → 2단계(신규 디자인 제작 워크플로우) → 3단계(단종 처리 + 담당자 관리 콘솔).
